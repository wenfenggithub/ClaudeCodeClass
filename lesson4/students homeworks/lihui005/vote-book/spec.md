# 图书投票应用 MVP 规格说明

## 1. 项目目标

本项目要实现一个网页版本的图书投票应用。用户可以在候选图书中查看图书信息，并在登录后为喜欢的图书投票。投票后，页面上的票数需要及时更新。

当前版本已完成基础 MVP、用户注册/登录、登录后投票、每人最多 3 票、管理员图书管理、书目自动补齐、本地真实封面、8 本候选图书、Apple 风格列表视图、投票占比进度条、排行榜 Top 3、实时最高票提示和管理员一键重置票数。后端继续使用 Node.js 技术，数据继续通过本地 JSON 文件存储，暂不引入数据库。

## 本地运行与访问地址

- 启动命令：`npm start`
- 默认访问地址：`http://127.0.0.1:3002`
- 默认端口使用 `3002`，避免和本机其他项目已占用的 `3000`、`3001` 冲突。
- 如需临时使用其他端口，可以通过环境变量覆盖，例如：`PORT=3010 npm start`。

## 2. MVP 功能范围

当前网页已包含以下功能：

- 展示候选图书列表，当前默认 8 本书。
- 每本书展示基础信息，包括书名、作者、简介和封面地址。
- 每本书展示当前票数。
- 未登录用户可以浏览图书列表、票数、排行榜和投票占比。
- 已登录用户可以点击每本书对应的“投票”按钮。
- 点击投票后，后端更新该书票数和当前用户投票记录。
- 投票成功后，前端刷新票数、剩余票数、排行榜、投票占比进度条和最高票提示。
- 当数据文件不存在或内容为空时，系统需要有明确的初始化或错误提示策略。

用户模块已包含以下功能：

- 用户可以通过用户名和密码注册。
- 用户可以通过用户名和密码登录。
- 用户可以退出登录。
- 页面可以展示当前登录用户。
- 未登录用户可以浏览图书列表和当前票数。
- 只有已登录用户可以投票。
- 未登录用户点击投票时，页面提示先登录。
- 用户密码不能明文存储，需要使用 Node.js 内置 `crypto` 进行加盐哈希。
- 用户信息存储在 `user.json`。
- 会话信息可以使用 `session.json` 或内存会话存储，具体实现需要在开发时说明选择理由。

管理员、书目和竞赛展示模块已包含以下功能：

- 必须登录才能投票，前端和后端都需要强制执行该规则。
- 未登录用户仍可以浏览图书列表和票数，但不能提交投票请求。
- 增加管理员角色，管理员可以调整候选图书内容。
- 管理员可以编辑图书的书名、作者、简介和封面地址。
- 管理员图书管理操作需要后端校验管理员权限。
- 候选图书数量从 5 本扩展到 8 本，即在原有 5 本书基础上新增 3 本书。
- 新增图书需要同步初始化投票数据，保证 `book.json` 和 `vote.json` 图书 ID 对齐。
- 第 4 项为页面布局调整：登录窗口放在页面右上角，页面左侧放一句有关读书的金句。
- 管理员调整书目时，输入或修改书名后，前端调用管理员专用接口，后端根据本地 `bookCatalog.json` 自动返回作者、简介和封面地址建议。
- 自动补齐只作用于管理员编辑表单，不自动写入 `book.json`，管理员确认并点击保存后才持久化。
- 每一本书的封面应使用对应真实书籍封面，优先使用本地 `/covers/` 图片，不再使用无关的通用书籍图片。
- 书籍目录页面的候选图书展示需要从画廊式或卡片网格排列调整为现代 SaaS 风格的纵向列表视图。
- 书籍目录布局调整只改变展示结构和视觉层级，需要保留现有图书数据、投票、登录、管理员编辑、书目自动补齐和本地封面等业务逻辑；如果页面已有搜索、筛选、排序、分页或点击跳转能力，也需要保持可用。
- 书籍目录列表项的票数区域需要增加投票占比进度条，用于展示该书票数占所有候选书总票数的百分比。
- 投票占比优先在前端基于当前 `GET /api/books` 返回的 `votes` 数据计算，不新增后端接口，不修改 `book.json` 或 `vote.json` 数据结构。
- 阶段二竞赛展示效果增强需要增加排行榜 Top 3、前三名排名标签、最高票提示强化和 0 票空状态优化。
- 排行榜、前三名标签、最高票提示和 0 票空状态优先在前端基于当前 `state.books` 或等价图书数组中的 `votes` 数据计算，不新增后端接口，不修改现有 JSON 数据结构。
- 阶段三竞赛体验优化需要调整投票规则、页面顶部结构、页面标题、阅读金句、Apple 风格视觉、管理员列表化管理和管理员一键重置票数能力。
- 每个登录用户在本次活动中最多可以投 3 票，3 票为用户总投票上限，不是每本书 3 票；前端需要显示剩余可投票数，后端必须校验投票上限。
- 为实现每人最多 3 票，后端需要记录用户投票次数或投票记录。建议新增 `userVote.json`，不修改 `book.json` 和 `user.json`；`vote.json` 继续存储每本书聚合票数。
- 首页主标题需要调整为“2026.5月阅读书目投票活动”。
- 首页金句需要替换为更强调阅读意义、成长价值和长期影响的文案。
- 页面整体视觉统一为 Apple 风格，顶部区域需要重新组织，要求紧凑、清晰、美观。
- 管理员图书管理区需要改为列表形式，和首页书籍列表保持一致的清晰信息层级。
- 管理员新增一键重置票数权限，可以将所有图书票数归零；该操作必须二次确认，并且只有管理员可以调用后端接口。

## 3. 暂不包含的功能

当前版本暂不包含以下功能：

- 数据库存储。
- 排行榜独立页面。
- 评论、收藏、分享等社交功能。
- 复杂的防刷票机制。
- 复杂权限体系和多级管理员角色。
- 长期安全会话、刷新令牌、多设备管理等完整认证体系。
- 同一用户对同一本书重复投票限制。
- 管理员后台独立应用。

这些能力可以在后续版本中逐步扩展。

## 4. 页面结构设计

应用首屏就是图书投票主页面，不需要单独的营销首页。

页面建议包含以下区域：

- 顶部活动区：展示英文标识“Reading Vote 2026”、活动标题“2026.5月阅读书目投票活动”、活动说明、阅读金句、登录或当前用户信息。
- 顶部标题区：主标题固定为“2026.5月阅读书目投票活动”，浏览器标题也使用同一文案。
- 阅读金句区：展示“阅读让我们在有限的时间里，抵达更辽阔的世界。”，作为页面首屏的内容引导。
- 用户状态区：未登录时展示注册和登录入口；已登录时展示当前用户名、角色、剩余可投票数和退出按钮。
- 简短说明区：说明用户可以为喜欢的图书投票，并提示每人最多可投 3 票。
- 状态提示区：默认显示“正在加载图书...”，加载后显示当前最高票或暂无投票状态，错误时显示明确失败原因。
- 图书目录列表区：以现代 SaaS 风格的纵向列表展示候选图书，不使用画廊式或网格卡片排列。
- 图书列表项：每本书作为一行列表项展示，包含封面缩略图、核心信息、票数和操作区。
- 排行榜 Top 3 区域：展示当前得票前三名图书，用于增强竞赛展示效果。
- 管理员图书管理区：仅管理员可见，用于调整图书内容。

页面顶部结构重整要求：

- 顶部区域需要从零散的左右分布调整为更紧凑、清晰、美观的首屏结构。
- 标题、活动说明、阅读金句、登录/用户信息、剩余投票数和状态提示之间需要形成明确层级。
- 桌面端建议使用紧凑的两栏或主次分区：左侧展示活动标题、说明和金句，右侧展示登录/用户信息和剩余投票数。
- 移动端应自然堆叠，避免标题、登录表单、状态栏或金句互相挤压。
- 顶部区域不应占用过高首屏空间，需要让排行榜和图书列表更早可见。
- 顶部状态提示应与投票规则、最高票提示和错误反馈保持可读，不应和登录表单混在一起。
- 保留注册、登录、当前用户、角色、退出登录等现有功能。
- 保留排行榜、图书列表、投票进度条、最高票提示等现有能力。

Apple 风格视觉要求：

- 页面整体继续使用 Apple 风格，而不是回到通用 SaaS 或营销页风格。
- 使用系统字体、浅色背景、充足但克制的留白、柔和阴影、细边框、半透明或浅色面板、清晰按钮状态和高级感排版。
- 主色可以延续 Apple 风格蓝色，辅助色需要克制，不应出现过多高饱和颜色。
- 页面卡片圆角、阴影和边框需要保持一致；不得使用杂乱渐变、大面积装饰图形或过强视觉噪声。
- 视觉重点应服务于竞赛展示：标题清晰、排行榜醒目、投票状态可快速判断、管理员区易编辑。
- 移动端和桌面端都需要保持清晰、紧凑和可读。

每个图书列表项建议包含：

- 图书封面缩略图。
- 图书名称。
- 作者。
- 简短简介。
- 图书 ID 标签，例如 `book-001`。
- “候选图书”状态标签。
- 未登录时显示“需要登录”，已登录时显示剩余票数或“投票已用完”。
- 当前领先图书显示“当前领先”标签。
- 排名前三且票数大于 0 的图书显示 `#1`、`#2`、`#3` 标签。
- 当前票数。
- 当前票数占所有候选书总票数的百分比。
- 投票占比进度条。
- 投票按钮。未登录时按钮文案为“登录后投票”，有票时为“投票”，投满后为“已达上限”。

图书目录列表区视觉要求：

- 采用 Apple 风格的纵向列表，整体应干净、克制、信息优先。
- 使用浅色背景、清晰分区、轻量边框或分隔线、规范按钮、状态标签和稳定间距。
- 桌面端每一行应有清晰的信息层级，例如左侧封面缩略图，中间书名、作者、简介和元信息，右侧票数和投票操作。
- 移动端列表项需要自适应窄屏，封面、文字和操作区不能溢出、重叠或难以点击。
- 页面应偏向 Apple 产品化工具界面，不应做成营销页、海报页、装饰性书店风格或大面积画廊展示。
- 布局调整不应改变现有数据结构、接口契约和业务流程。

投票占比进度条要求：

- 每本书的票数区域旁边或下方展示进度条。
- 每本书同时展示当前票数、占比百分比和对应进度条。
- 占比计算公式为：当前书籍票数 / 所有候选书总票数。
- 百分比基于前端当前 `state.books` 或等价图书数组中的 `votes` 字段计算。
- 当所有候选书总票数为 `0` 时，每本书占比显示为 `0%`，进度条宽度为 `0%`。
- 前端需要避免出现除以 0、`NaN`、`Infinity` 或错误进度条宽度。
- 用户投票成功后，前端需要重新计算所有候选书的占比，并同步更新所有进度条，而不是只更新当前图书的数字票数。
- 进度条视觉应符合现代 SaaS 风格：轻量背景轨道、清晰填充色、克制动效或无动效。
- 移动端和桌面端都需要正常显示，票数、百分比、进度条和投票按钮不能重叠或溢出。
- 进度条不改变投票接口、管理员管理接口、书目自动补齐接口和数据文件结构。

投票上限要求：

- 每个登录用户在本次活动中最多可以投 3 票。
- 3 票是用户总投票上限，不是每本书各 3 票。
- 用户可以把 3 票投给同一本书，也可以分散投给不同书；如果后续要限制同一本书重复投票，需要另行定义。
- 未登录用户仍然可以浏览图书、票数、排行榜和进度条，但不能投票。
- 前端需要在用户区域或顶部活动区展示剩余可投票数，例如“剩余 2 票”。
- 每次投票成功后，前端需要同步更新当前用户已投票数、剩余票数、图书票数、排行榜、进度条和最高票提示。
- 当用户已投满 3 票时，页面投票按钮应禁用或点击后提示“已达到投票上限”。
- 后端必须在 `POST /api/votes` 中校验当前用户已投票数，不能只依赖前端限制。
- 如果用户已投满 3 票后继续请求投票接口，后端返回明确错误，例如“已达到投票上限”。
- 用户退出登录后，页面不显示剩余投票数；再次登录后需要重新从后端获取该用户的投票状态。
- 管理员一键重置所有票数后，用户投票记录也需要同步清空，使所有用户重新拥有 3 票。

排行榜 Top 3 区域要求：

- 排行榜展示当前得票前三名图书。
- 每个排行项需要展示排名、书名、票数和投票占比。
- 排行榜基于当前前端拿到的图书 `votes` 数据计算。
- 排行榜应和书籍目录使用同一份前端数据来源，避免出现列表和排行榜不一致。
- 投票成功后，排行榜需要实时更新排名、票数和占比。
- 排行榜不新增后端接口，不修改 `book.json`、`vote.json` 或其他数据文件结构。
- 排行榜应保持现代 SaaS 风格，适合竞赛现场快速展示当前结果。
- 移动端和桌面端都需要正常显示，不应挤压登录区、状态栏或图书列表。

排行榜排序和并列规则：

- 优先按票数从高到低排序。
- 如果票数相同，按图书在当前候选书列表中的原始顺序排序，保证展示稳定。
- 总票数为 `0` 时，不展示虚假的前三名结果，应显示友好的空状态，例如“暂无投票，成为第一个投票的人”。
- 如果候选书不足 3 本，只展示已有候选书的排行项。

前三名标签要求：

- 图书列表中的前三名可以显示 `#1`、`#2`、`#3` 排名标签。
- 排名标签应基于和排行榜相同的排序结果计算。
- 票数为 `0` 时不应显示前三名标签，避免把未投票结果误导为排行结果。
- 当前最高票提示应显示真正得票最高的书，而不是刚刚投票的书。
- 如果存在并列最高票，最高票提示可以显示多个并列书名。

0 票空状态要求：

- 当所有候选书总票数为 `0` 时，排行榜区域显示友好空状态。
- 空状态文案建议为“暂无投票，成为第一个投票的人”。
- 图书列表中的投票占比和进度条继续显示 `0%` 和 `0%` 宽度。
- 页面不能出现 `NaN`、`Infinity` 或空白排行项。

注册和登录区域建议包含：

- 位于页面头部右侧的登录注册面板。
- 用户名输入框。
- 密码输入框。
- 注册按钮。
- 登录按钮。
- 登录成功后隐藏登录表单，展示当前用户名。
- 当前角色标签，管理员显示“管理员”，普通用户显示“普通用户”。
- 当前投票额度，格式为“剩余 N 票 / 共 3 票，已投 M 票”。
- 退出登录按钮。
- 注册、登录、退出失败时的错误提示。

管理员图书管理区域建议包含：

- 当前 8 本候选图书的管理列表，必须使用列表形式，不使用卡片网格或画廊式排列。
- 管理区标题为“图书管理”，说明文案为“调整候选图书的书名、作者、简介和封面地址。”。
- 每本书作为一行管理表单展示，表头显示当前书名、图书 ID 和保存按钮。
- 每一行管理表单包含书名、作者、简介、封面地址和保存按钮。
- 书名输入框触发自动补齐，补齐作者、简介和封面地址。
- 自动补齐状态提示，例如“正在查找书目信息...”“已补齐《书名》的信息。”“未找到匹配书目信息”。
- 保存按钮。
- 保存成功或失败提示。
- “一键重置票数”按钮，仅管理员可见。
- 重置票数操作前必须有二次确认，当前确认文案为“确认将所有图书票数归零吗？该操作会清空所有用户投票记录。”。
- 重置成功后，首页票数、排行榜、投票占比进度条、最高票提示和剩余投票数都需要同步刷新。
- 普通用户和未登录用户不可见或不可操作该区域。

管理员列表化视觉要求：

- 管理员管理区应和首页图书列表保持一致的 Apple 风格：浅色面板、细边框、稳定间距、清晰分组。
- 每一行管理项需要易于扫描和编辑，字段排列应紧凑但不拥挤。
- 桌面端可使用多列布局组织输入框；移动端需要纵向堆叠，输入框和按钮不能重叠或溢出。
- 书名、作者、简介和封面地址输入区需要保留自动补齐状态提示。
- 保存按钮和重置票数按钮需要有明确视觉层级；重置票数属于高风险操作，应使用危险或警示样式。

左侧读书金句区域建议：

- 放在页面顶部活动区中，与标题和登录状态形成清晰布局。
- 文案建议替换为“阅读让我们在有限的时间里，抵达更辽阔的世界。”。
- 金句需要强调阅读的意义、成长价值和长期影响，避免空泛口号。
- 在移动端可以折叠到登录区域上方或页面标题下方，避免和登录表单重叠。
- 金句只作为页面内容氛围和视觉引导，不影响图书列表和投票主流程。

页面设计应采用 Apple 风格，保持简洁、清晰、紧凑和可扫描，优先保证用户能快速浏览图书并完成投票。

## 5. 用户操作流程

基础流程：

1. 用户打开网页。
2. 前端请求后端接口，获取当前登录用户信息。
3. 前端请求后端接口，获取图书列表和当前票数。
4. 页面展示所有候选图书和登录状态。
5. 未登录用户可以浏览图书和票数。
6. 用户注册或登录成功后，页面展示当前用户名。
7. 已登录用户点击某本书的“投票”按钮。
8. 前端向后端提交投票请求。
9. 后端校验登录状态和图书是否存在。
10. 后端更新 `vote.json` 中对应图书的票数。
11. 后端返回最新投票结果。
12. 前端更新页面中的票数。
13. 页面展示投票成功提示。
14. 如果用户已投满 3 票，前端提示已达到投票上限，后端拒绝继续投票。

管理员流程：

1. 管理员登录。
2. 前端通过当前用户信息识别管理员角色。
3. 页面展示管理员图书管理区域。
4. 管理员修改图书的书名、作者、简介或封面地址。
5. 前端向后端提交图书更新请求。
6. 后端校验当前用户是否为管理员。
7. 后端校验图书是否存在和字段是否合法。
8. 后端更新 `book.json`。
9. 后端返回更新后的图书信息。
10. 前端刷新图书列表和管理区内容。
11. 管理员如需将所有票数归零，点击一键重置票数按钮。
12. 前端弹出二次确认，管理员确认后发送重置请求。
13. 后端校验管理员权限，将所有图书票数归零，并清空用户投票记录。
14. 前端刷新图书列表、排行榜、进度条、最高票提示和剩余投票数。

异常流程：

1. 如果图书数据加载失败，页面显示错误提示。
2. 如果投票请求失败，页面提示用户稍后重试。
3. 如果未登录用户点击投票，页面提示先登录。
4. 如果用户投票的图书不存在，后端返回错误信息。
5. 如果注册用户名已存在，后端返回明确错误信息。
6. 如果登录用户名或密码错误，后端返回明确错误信息。
7. 如果普通用户调用管理员接口，后端返回无权限错误。
8. 如果管理员更新不存在的图书，后端返回图书不存在。
9. 如果管理员提交无效图书字段，后端返回参数错误。
10. 如果用户投票已达到 3 票，后端返回已达到投票上限。
11. 如果普通用户或未登录用户调用重置票数接口，后端返回无权限错误或请先登录。
12. 如果管理员取消重置票数二次确认，前端不发送重置请求。
13. 如果文件读写失败，后端返回服务器错误信息。

## 6. 数据文件设计

### 6.1 `book.json`

`book.json` 用于存储图书基础信息。该文件不直接存储票数，避免图书基础信息和投票数据混在一起。

建议数据结构：

```json
[
  {
    "id": "book-001",
    "title": "示例图书一",
    "author": "作者 A",
    "description": "这是一本示例图书的简介。",
    "coverUrl": "/covers/book-001.jpg"
  }
]
```

字段说明：

| 字段 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| `id` | string | 是 | 图书唯一标识，用于投票关联 |
| `title` | string | 是 | 图书名称 |
| `author` | string | 是 | 作者名称 |
| `description` | string | 是 | 图书简介 |
| `coverUrl` | string | 否 | 图书封面地址，推荐使用真实书籍封面路径 |

封面说明：

- 推荐将真实书籍封面图片放在 `public/covers/` 目录中。
- `coverUrl` 使用相对路径，例如 `/covers/book-001.jpg`。
- 封面文件建议按图书 ID 命名，例如 `public/covers/book-001.jpg`，便于 `book.json`、`vote.json` 和封面资源对齐。
- 管理员也可以填写外部封面 URL，但公开上线时需要注意图片版权、稳定性和防盗链问题。
- 如果暂时没有真实封面，可以临时使用明确标识的占位图，但任务验收前不应继续使用 Unsplash 等无关通用图片。

### 6.2 `vote.json`

`vote.json` 用于存储每本书的聚合投票数。用户级投票次数不存放在该文件中，避免聚合票数和用户行为记录混在一起。

建议数据结构：

```json
{
  "book-001": {
    "bookId": "book-001",
    "count": 0
  }
}
```

字段说明：

| 字段 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| `bookId` | string | 是 | 对应 `book.json` 中的图书 `id` |
| `count` | number | 是 | 当前图书获得的票数 |

说明：

- `vote.json` 使用图书 `id` 作为 key，方便快速根据图书查找票数。
- 如果某本书在 `book.json` 中存在，但 `vote.json` 中没有对应记录，后端可以默认该书票数为 `0`。
- 投票成功后，只更新 `vote.json`，不修改 `book.json`。
- 管理员一键重置票数时，将 `vote.json` 中所有候选图书的 `count` 设置为 `0`。

### 6.3 `userVote.json`

`userVote.json` 用于记录每个用户在本次活动中的投票次数和投票明细，从而支持“每人最多投 3 票”的后端校验。

建议数据结构：

```json
{
  "user-001": {
    "userId": "user-001",
    "total": 2,
    "votes": [
      {
        "bookId": "book-001",
        "createdAt": "2026-05-14T00:00:00.000Z"
      },
      {
        "bookId": "book-006",
        "createdAt": "2026-05-14T00:01:00.000Z"
      }
    ]
  }
}
```

字段说明：

| 字段 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| `userId` | string | 是 | 对应 `user.json` 中的用户 `id` |
| `total` | number | 是 | 当前用户在本次活动中的已投票数 |
| `votes` | object[] | 是 | 当前用户的投票明细 |
| `votes[].bookId` | string | 是 | 被投票图书 ID |
| `votes[].createdAt` | string | 是 | 投票时间，建议使用 ISO 8601 字符串 |

说明：

- 新增 `userVote.json` 是为了实现后端投票上限校验；仅靠 `vote.json` 中的图书总票数无法判断某个用户已经投了几票。
- `total` 应与 `votes.length` 保持一致；如果出现不一致，后端可以以 `votes.length` 为准重新计算，避免数据错误导致用户无法投票。
- 投票成功时，需要同时更新 `vote.json` 中对应图书的聚合票数，以及 `userVote.json` 中当前用户的投票记录。
- 如果 `userVote.json` 不存在，后端可以初始化为空对象。
- 管理员一键重置票数时，需要同步清空 `userVote.json`，让所有用户重新拥有 3 票。
- `userVote.json` 不存储用户名、密码、密码哈希或其他敏感信息。

### 6.4 `user.json`

`user.json` 用于存储注册用户信息。密码不能明文存储，需要保存加盐哈希结果。

建议数据结构：

```json
[
  {
    "id": "user-001",
    "username": "demo",
    "email": "demo@example.com",
    "passwordHash": "hashed-password",
    "salt": "random-salt",
    "createdAt": "2026-05-12T00:00:00.000Z",
    "role": "user"
  }
]
```

字段说明：

| 字段 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| `id` | string | 是 | 用户唯一标识 |
| `username` | string | 是 | 登录用户名，需要唯一 |
| `email` | string | 否 | 用户邮箱，可用于密码找回；未绑定时为空字符串。同一邮箱不可被多个账号使用 |
| `passwordHash` | string | 是 | 使用 Node.js 内置 `crypto` 生成的密码哈希 |
| `salt` | string | 是 | 每个用户独立的随机盐 |
| `createdAt` | string | 是 | 用户创建时间，建议使用 ISO 8601 字符串 |
| `role` | string | 是 | 用户角色，普通用户为 `user`，管理员为 `admin` |

说明：

- `user.json` 不允许保存明文密码。
- 用户名需要去除首尾空格后再校验和存储。
- MVP 阶段不要求密码复杂度规则，但用户名和密码不能为空。
- 注册时需要检查用户名是否已经存在。
- 普通注册用户默认角色为 `user`。
- 管理员账号可以通过初始化数据或后续管理脚本创建，MVP 阶段不提供公开注册管理员能力。

### 6.5 会话数据

基础用户模块需要保存登录状态。可以选择以下方案之一：

- 内存会话：实现简单，服务重启后登录状态丢失，适合 MVP 阶段快速验证。
- `session.json` 文件会话：服务重启后会话仍可保留，但需要额外处理过期、清理和文件写入。

建议优先使用内存会话，以保持实现简单，并在文档或代码注释中说明服务重启后需要重新登录。如果选择文件会话，需要新增 `session.json` 并避免保存用户密码或敏感信息。

### 6.6 `bookCatalog.json`

`bookCatalog.json` 用于存储可自动补齐的本地书目信息。管理员输入书名后，后端从该文件中查找匹配记录，并返回作者、简介和封面地址建议。

本阶段采用“本地真实封面固定绑定 + 本地书目库自动补齐”的组合方案：

- 方案一：每本候选书都通过 `bookId` 绑定一个本地真实封面文件，封面统一存放在 `public/covers/`。
- 方案二：`bookCatalog.json` 维护书名、别名、作者、简介和封面路径映射，管理员输入书名时自动补齐这些字段。
- `book.json` 负责当前页面实际展示的数据，`bookCatalog.json` 负责管理员编辑时的建议数据。
- 同一本书在 `book.json` 和 `bookCatalog.json` 中必须使用同一个 `bookId` 和同一个 `coverUrl`，避免书名和图片再次错配。

建议数据结构：

```json
[
  {
    "bookId": "book-001",
    "title": "人类简史",
    "aliases": ["Sapiens"],
    "author": "尤瓦尔·赫拉利",
    "description": "从认知革命到现代社会，梳理人类文明演化的关键路径。",
    "coverUrl": "/covers/book-001.jpg",
    "coverSource": "本地封面资源"
  }
]
```

字段说明：

| 字段 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| `bookId` | string | 否 | 如果该书对应当前候选图书，可填 `book.json` 中相同的图书 ID |
| `title` | string | 是 | 图书名称，用于匹配管理员输入 |
| `aliases` | string[] | 否 | 图书别名、英文名或常见译名，用于提升匹配成功率 |
| `author` | string | 是 | 作者名称 |
| `description` | string | 是 | 图书简介 |
| `coverUrl` | string | 否 | 真实书籍封面地址，优先使用本地 `/covers/` 路径 |
| `coverSource` | string | 否 | 封面来源说明，便于后续确认版权和可用性 |

说明：

- MVP 阶段优先使用本地书目库，不依赖外部图书 API。
- `bookCatalog.json` 至少需要覆盖当前 8 本候选图书。
- 匹配方式优先支持去除首尾空格后的书名精确匹配，其次支持 `aliases` 精确匹配；英文别名可以忽略大小写。
- 未匹配到书名时返回空建议，不覆盖管理员已经输入的内容。
- 后续可以扩展为模糊搜索、拼音搜索或接入外部图书数据 API。
- 自动补齐结果只作为表单建议，不应绕过管理员确认和保存操作。

### 6.7 真实封面资源

真实封面资源用于保证页面中的每一本书图片都与对应书籍一致。

资源规则：

- 优先将封面图片保存到 `public/covers/`。
- 当前 8 本候选书建议使用 `/covers/book-001.jpg` 到 `/covers/book-008.jpg` 的固定路径。
- 封面文件名与 `bookId` 一一对应，例如 `book-001` 对应 `public/covers/book-001.jpg`。
- 页面展示时以 `book.json` 中的 `coverUrl` 为准，不根据书名临时猜测封面。
- 管理员自动补齐时以 `bookCatalog.json` 中的 `coverUrl` 为准，补齐后仍需管理员点击保存才会写入 `book.json`。
- `book.json` 中的 `coverUrl` 需要指向对应真实封面路径。
- `bookCatalog.json` 中的 `coverUrl` 应与对应真实封面路径保持一致，便于管理员自动补齐。
- 不应使用与具体书籍无关的通用书本、书架或阅读场景图片。
- 如果使用外部图片 URL，需要确认该 URL 稳定可访问，并在 `coverSource` 中记录来源。
- 如果真实封面资源暂时缺失，应在 `task.md` 中标记为阻塞或未完成，而不是用无关图片冒充真实封面。

一致性规则：

- `book.json`、`bookCatalog.json` 和 `public/covers/` 之间以 `bookId` 作为主关联键。
- 更新书名时，如果 `bookCatalog.json` 匹配到记录，需要同时补齐作者、简介和封面路径，降低人工填错概率。
- 更新封面时，需要同时检查 `book.json` 和 `bookCatalog.json`，保证两个文件中的 `coverUrl` 一致。
- 验收时需要逐本检查 8 本书的书名、作者和封面是否对应同一本真实书籍。

### 6.8 `passwordReset.json`

`passwordReset.json` 用于存储密码重置令牌的 SHA256 哈希，不保存原始令牌。

建议数据结构：

```json
[
  {
    "tokenHash": "sha256-hash-of-reset-token",
    "userId": "user-admin",
    "createdAt": "2026-05-22T00:00:00.000Z",
    "expiresAt": "2026-05-22T00:30:00.000Z",
    "usedAt": null
  }
]
```

字段说明：

| 字段 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| `tokenHash` | string | 是 | 重置令牌的 SHA256 哈希，原始令牌仅在邮件中明文出现 |
| `userId` | string | 是 | 对应 `user.json` 中的用户 `id` |
| `createdAt` | string | 是 | 令牌创建时间 |
| `expiresAt` | string | 是 | 令牌过期时间，默认 30 分钟有效 |
| `usedAt` | string \| null | 是 | 令牌使用时间，未使用时为 `null` |

说明：

- 每次读取时自动删除已过期或已使用超过 24 小时的记录。
- 重置密码成功后，同一用户所有未使用令牌一并标记为 `usedAt`，防止旧链接重放。
- 文件不存在时初始化为空数组。

## 7. 后端接口设计思路

后端使用 Node.js 提供 HTTP API。当前实现可以继续使用 Node.js 原生 HTTP 模块，不要求引入第三方依赖。新增用户模块时需要保持现有图书和投票接口兼容。

### 7.1 获取图书列表

接口：

```http
GET /api/books
```

用途：

- 获取图书基础信息。
- 合并当前票数后返回给前端。

响应示例：

```json
{
  "success": true,
  "data": [
    {
      "id": "book-001",
      "title": "示例图书一",
      "author": "作者 A",
      "description": "这是一本示例图书的简介。",
      "coverUrl": "/images/book-001.jpg",
      "votes": 12
    }
  ]
}
```

### 7.2 获取投票结果

接口：

```http
GET /api/votes
```

用途：

- 获取全部图书的当前投票结果。
- 可用于页面定时刷新票数或投票后重新同步数据。

响应示例：

```json
{
  "success": true,
  "data": {
    "book-001": {
      "bookId": "book-001",
      "count": 12
    }
  }
}
```

### 7.3 提交投票

接口：

```http
POST /api/votes
```

请求体：

```json
{
  "bookId": "book-001"
}
```

用途：

- 为指定图书增加 1 票。
- 需要用户已登录。
- 校验当前用户本次活动已投票数是否小于 3。
- 更新 `vote.json` 中对应图书的聚合票数。
- 更新 `userVote.json` 中当前用户的投票记录。
- 返回该书最新票数，以及当前用户已投票数和剩余票数。

响应示例：

```json
{
  "success": true,
  "data": {
    "bookId": "book-001",
    "count": 13,
    "userVoteTotal": 1,
    "remainingVotes": 2,
    "maxVotesPerUser": 3
  },
  "message": "投票成功"
}
```

错误响应示例：

```json
{
  "success": false,
  "message": "图书不存在"
}
```

未登录错误响应示例：

```json
{
  "success": false,
  "message": "请先登录后再投票"
}
```

达到投票上限错误响应示例：

```json
{
  "success": false,
  "message": "已达到投票上限",
  "data": {
    "userVoteTotal": 3,
    "remainingVotes": 0,
    "maxVotesPerUser": 3
  }
}
```

### 7.4 用户注册

接口：

```http
POST /api/register
```

请求体：

```json
{
  "username": "demo",
  "password": "password123",
  "email": "demo@example.com"
}
```

用途：

- 创建新用户。
- 校验用户名、密码和邮箱不能为空。
- 校验邮箱格式正确且未被其他账号使用。
- 校验用户名不能重复。
- 使用 Node.js 内置 `crypto` 生成 `salt` 和 `passwordHash`。
- 将用户信息（含邮箱）写入 `user.json`。

响应示例：

```json
{
  "success": true,
  "data": {
    "id": "user-001",
    "username": "demo"
  },
  "message": "注册成功"
}
```

## 8. 前后端交互流程

页面初始化：

1. 前端加载页面资源。
2. 前端请求 `GET /api/me`。
3. 前端根据返回结果展示未登录或已登录状态；已登录时展示当前用户已投票数和剩余可投票数。
4. 前端请求 `GET /api/books`。
5. 后端读取 `book.json` 和 `vote.json`。
6. 后端将图书信息和票数合并。
7. 前端渲染顶部活动区、排行榜、图书列表和投票占比进度条。

注册：

1. 用户输入用户名和密码。
2. 前端发送 `POST /api/register`。
3. 后端校验参数和用户名唯一性。
4. 后端生成 `salt` 和 `passwordHash`。
5. 后端写入 `user.json`。
6. 前端展示注册成功提示。

登录：

1. 用户输入用户名和密码。
2. 前端发送 `POST /api/login`。
3. 后端校验用户名和密码。
4. 后端创建会话。
5. 前端展示当前用户名和退出按钮。

退出登录：

1. 用户点击退出登录按钮。
2. 前端发送 `POST /api/logout`。
3. 后端清除会话。
4. 前端恢复未登录状态。

管理员更新图书：

1. 管理员登录。
2. 前端请求 `GET /api/me`，确认当前用户角色为 `admin`。
3. 前端展示图书管理区。
4. 管理员修改图书内容并点击保存。
5. 前端发送 `PUT /api/books/:bookId`。
6. 后端校验登录状态和管理员角色。
7. 后端读取 `book.json`，查找对应图书。
8. 后端校验并更新允许修改的字段。
9. 后端写回 `book.json`。
10. 前端刷新图书列表。

管理员自动补齐书目信息：

1. 管理员登录。
2. 管理员在图书管理表单中输入或修改书名。
3. 前端在输入停止后或输入框失焦时请求 `GET /api/book-suggestions?title=...`。
4. 后端校验管理员权限。
5. 后端读取 `bookCatalog.json` 并查找匹配书名。
6. 如果匹配成功，返回作者、简介和封面地址建议。
7. 如果匹配失败，返回 `data: null`，前端不覆盖当前表单内容。
8. 前端将匹配建议自动填入作者、简介和封面地址字段，并展示补齐状态。
9. 管理员检查内容并点击保存。
10. 后端通过 `PUT /api/books/:bookId` 更新 `book.json`。
11. 如果管理员没有点击保存，刷新页面后 `book.json` 内容不应发生变化。

真实封面维护：

1. 确认当前 8 本候选书的真实封面来源。
2. 将封面图片保存到 `public/covers/`。
3. 使用 `book-001.jpg` 到 `book-008.jpg` 作为本地封面文件名，并与 `bookId` 一一对应。
4. 更新 `book.json` 中每本书的 `coverUrl`。
5. 更新 `bookCatalog.json` 中对应书目的 `bookId` 和 `coverUrl`。
6. 对比 `book.json` 与 `bookCatalog.json`，确认同一本书的 `coverUrl` 一致。
7. 打开页面确认每本书显示的是对应真实书籍封面。

提交投票：

1. 用户点击图书列表项上的投票按钮。
2. 如果前端判断未登录，提示用户先登录，不发送投票请求。
3. 如果前端判断当前用户剩余票数为 `0`，提示“已达到投票上限”，不发送投票请求。
4. 如果已登录且有剩余票数，前端禁用当前按钮，避免短时间重复点击造成并发请求。
5. 前端发送 `POST /api/votes`，请求体包含 `bookId`。
6. 后端检查当前请求是否已登录。
7. 后端检查 `bookId` 是否存在于 `book.json`。
8. 后端读取 `userVote.json`，检查当前用户已投票数是否小于 `3`。
9. 如果当前用户已投满 3 票，后端返回“已达到投票上限”，不修改任何票数。
10. 如果未达到上限，后端读取 `vote.json`，将对应图书票数加 1。
11. 后端更新 `userVote.json` 中当前用户的投票记录。
12. 后端写回 `vote.json` 和 `userVote.json`。
13. 后端返回最新票数、当前用户已投票数和剩余票数。
14. 前端更新当前图书的票数、用户剩余票数和投票按钮状态。
15. 前端基于所有候选书最新票数重新计算每本书的投票占比。
16. 前端同步更新每本书的百分比和进度条宽度。
17. 前端基于最新票数重新计算排行榜 Top 3、前三名标签和最高票提示。
18. 前端恢复按钮状态，并展示当前最高票提示或投票成功反馈。

管理员一键重置票数：

1. 管理员登录。
2. 前端请求 `GET /api/me`，确认当前用户角色为 `admin`。
3. 前端在管理员管理区展示“一键重置票数”按钮。
4. 管理员点击按钮后，前端弹出二次确认。
5. 如果管理员取消确认，前端不发送请求。
6. 如果管理员确认，前端发送 `POST /api/votes/reset`。
7. 后端校验登录状态和管理员角色。
8. 后端读取当前候选图书列表，并将 `vote.json` 中所有候选书 `count` 设置为 `0`。
9. 后端清空 `userVote.json`。
10. 后端写回 `vote.json` 和 `userVote.json`。
11. 前端重新加载 `GET /api/me` 和 `GET /api/books`，刷新剩余票数、图书列表、排行榜、进度条和最高票提示。
12. 页面展示重置成功提示；排行榜进入 0 票空状态。

## 9. 文件存储方案说明

MVP 阶段使用本地 JSON 文件存储数据：

- `book.json` 存储相对稳定的图书基础信息，当前包含 8 本候选图书。
- `vote.json` 存储经常变化的投票数据。
- `userVote.json` 存储用户在本次活动中的投票次数和投票明细，用于校验每人最多 3 票。
- `user.json` 存储用户账号信息，只保存密码哈希和盐，不保存明文密码。
- `bookCatalog.json` 存储书名到作者、简介和真实封面的补齐建议。
- `passwordReset.json` 存储密码重置令牌的 SHA256 哈希，不保存原始令牌。
- `public/covers/` 存储本地真实书籍封面图片。
- `.env` 存储 SMTP 凭据和运行配置（不入库）；`.env.example` 为模板文件。
- 会话使用内存存储，服务重启后需重新登录。

文件读写策略：

- 服务启动时可以读取 `book.json` 验证基础数据是否存在。
- 新增 3 本书后，需要为新增图书同步初始化 `vote.json` 中的投票记录。
- 每次获取图书列表时读取最新 `vote.json`，确保票数是最新的。
- 每次提交投票时读取、修改并写回 `vote.json`。
- 每次提交投票时读取、校验、修改并写回 `userVote.json`。
- 每次获取当前用户时，可以读取 `userVote.json` 计算当前用户已投票数和剩余票数。
- 每次管理员重置票数时，将 `vote.json` 中所有候选图书票数归零，并清空 `userVote.json`。
- 每次管理员更新图书内容时读取、修改并写回 `book.json`。
- 每次管理员请求书目信息建议时读取 `bookCatalog.json`。
- 更换真实封面时，需要同步更新 `book.json` 和 `bookCatalog.json` 中的 `coverUrl`。
- 每次注册时读取、修改并写回 `user.json`。
- 每次登录时读取 `user.json`，使用相同哈希算法校验密码。
- 写文件时需要使用格式化 JSON，方便人工查看和调试。

并发注意事项：

- 简单 MVP 中，可以先采用同步或串行化的文件读写方式降低实现复杂度。
- 如果同时有多个用户高频投票，直接读写 JSON 文件可能出现并发写入覆盖问题。
- 后续可以引入写入队列、文件锁、数据库或缓存系统解决并发一致性问题。

## 10. 异常情况处理

需要考虑以下异常：

- `book.json` 不存在：后端返回明确错误，提示图书数据未初始化。
- `book.json` 格式错误：后端返回服务器错误，并在日志中记录解析失败原因。
- `vote.json` 不存在：后端可以自动创建空投票文件，或将所有图书票数默认视为 `0`。
- `vote.json` 格式错误：后端返回服务器错误，避免覆盖已有异常数据。
- `userVote.json` 不存在：后端可以自动创建空对象。
- `userVote.json` 格式错误：后端返回服务器错误，避免错误放行或错误拦截用户投票。
- `userVote.json` 中当前用户记录缺失：视为该用户已投票数为 `0`。
- `userVote.json` 中 `total` 与 `votes.length` 不一致：后端可以以 `votes.length` 重新计算实际已投票数。
- `user.json` 不存在：后端可以自动创建空用户数组。
- `user.json` 格式错误：后端返回服务器错误，避免覆盖已有异常数据。
- 请求中的 `bookId` 为空：返回参数错误。
- 请求中的 `bookId` 不存在：返回图书不存在。
- 更新图书时未登录：返回请先登录。
- 更新图书时非管理员：返回无管理员权限。
- 更新图书时图书不存在：返回图书不存在。
- 更新图书时书名、作者、简介为空：返回参数错误。
- 更新图书时封面地址为空或格式不符合预期：返回参数错误或允许为空，具体规则需实现时保持一致。
- 查询书目信息建议时未登录：返回请先登录。
- 查询书目信息建议时非管理员：返回无管理员权限。
- 查询书目信息建议时书名为空：返回参数错误。
- 查询书目信息建议时未匹配：返回 `data: null`，前端保持原表单内容。
- `bookCatalog.json` 不存在：后端可以返回空建议，或创建空数组。
- `bookCatalog.json` 格式错误：后端返回服务器错误，避免返回错误补齐内容。
- 真实封面文件缺失：页面应显示明确的图片加载失败或占位状态，开发验收不得标记为真实封面完成。
- 外部封面 URL 不可访问：应更换为本地封面或稳定可访问的 URL。
- 注册用户名为空：返回参数错误。
- 注册密码为空：返回参数错误。
- 注册用户名已存在：返回用户名已存在。
- 登录用户名为空或密码为空：返回参数错误。
- 登录用户名不存在或密码错误：返回用户名或密码错误。
- 未登录用户提交投票：返回请先登录后再投票。
- 已登录用户投满 3 票后继续提交投票：返回已达到投票上限，不修改 `vote.json` 和 `userVote.json`。
- 重置票数时未登录：返回请先登录。
- 重置票数时非管理员：返回无管理员权限。
- 重置票数请求未确认或缺少确认字段：后端可以返回参数错误，前端仍必须提供二次确认。
- 文件写入失败：返回投票失败，并记录错误日志。
- 前端请求失败：页面显示错误提示，用户可以重试。

## 11. 后续可扩展方向

后续版本可以考虑以下扩展：

- 会话增强：支持会话过期、长期登录、多设备管理等能力。
- 同书重复投票限制：限制同一用户不能对同一本书重复投票。
- 数据库：将 JSON 文件替换为 SQLite、PostgreSQL、MySQL 或 MongoDB。
- 后台管理增强：支持独立管理后台和管理员账号管理。
- 书目信息增强：支持外部图书 API、模糊搜索、ISBN 查询和封面上传。
- 投票时间窗口：设置投票开始时间和截止时间。
- 投票记录：记录每次投票的时间、用户和图书。
- 实时更新：通过 WebSocket 或 Server-Sent Events 实时同步票数。
- 防刷票：加入验证码、频率限制和异常行为检测。
- 单元测试和接口测试：保障投票逻辑和文件读写逻辑稳定。

## 12. 验收标准

基础 MVP 已满足以下标准：

- 页面可以正常展示 8 本候选图书。
- 每本书可以展示书名、作者、简介、封面和当前票数。
- 基础 MVP 初始阶段曾支持用户直接点击投票按钮，对应图书票数增加 1。
- 基础 MVP 初始阶段投票成功后，页面上的票数会更新。
- 后端使用 Node.js 提供接口。
- 图书基础信息存储在 `book.json`。
- 投票信息存储在 `vote.json`。
- 基础 MVP 初始阶段不需要登录即可完成投票；当前版本已调整为必须登录才能投票。
- 当请求参数错误或图书不存在时，后端返回明确错误。
- 当数据文件读取或写入失败时，系统不会静默失败。
- 文档能作为后续开发的直接依据。

基础用户模块已满足以下标准：

- 用户可以使用用户名和密码注册。
- 用户名为空或密码为空时，注册失败并提示明确错误。
- 重复用户名注册失败并提示明确错误。
- `user.json` 中保存用户信息，且不包含明文密码。
- 正确用户名和密码可以登录。
- 错误密码登录失败并提示明确错误。
- 登录成功后页面显示当前用户名。
- 用户可以退出登录。
- 退出登录后页面恢复未登录状态。
- 未登录用户可以浏览图书和票数。

## 13. 新建图书自动抓取与邮箱密码找回（阶段四）

### 13.1 目标
- 管理员可以在图书管理区新建图书，填好书名和作者后从豆瓣自动抓取封面和简介，封面下载到 `public/covers/<bookId>.jpg` 本地保存。
- 所有用户都可以通过邮箱找回密码，重置链接通过 SMTP 邮件发出，30 分钟内有效，且仅可使用一次。

### 13.2 数据结构变更
- `user.json` 每个用户记录新增可选 `email` 字段，类型 `string`，未绑定时为空字符串 `""`；同一邮箱不可被多个账号同时使用。
- 新增 `data/passwordReset.json`，结构为数组，每个元素 `{tokenHash, userId, createdAt, expiresAt, usedAt}`；token 以 sha256 哈希存储，使用过或过期超过 24 小时的记录会在下一次读取时自动清理。
- `package.json` 新增运行时依赖 `nodemailer`。
- 新增 `.env.example` 示意 SMTP 相关环境变量，实际 `.env` 已在 `.gitignore` 中。

### 13.3 接口设计
- `POST /api/books`（管理员）：创建新图书。请求体 `{title, author, description, coverSourceUrl}`；服务器自动分配下一个 `book-NNN` ID，若 `coverSourceUrl` 是 `http(s)://` 则下载到 `public/covers/<bookId>.jpg`，若是 `/covers/...` 则直接使用，缺省使用 `/covers/cover-missing.jpg`；写入 `book.json` 和 `vote.json`（初始 `count: 0`）。
- `POST /api/book-enrich`（管理员）：根据 `{title, author}` 搜索豆瓣（`douban.com/search?cat=1001`），从搜索结果提取第一条图书详情页 URL，解析详情页获取书名、作者、简介和封面大图地址。返回 `{title, author, description, coverSourceUrl, source: "douban"}`；豆瓣未匹配或解析失败返回 `data: null`。
- `POST /api/forgot-password`（公开）：请求体 `{email}`，无论邮箱是否存在都返回统一文案，不泄漏存在性；邮箱存在时生成 32 字节 token、记录哈希到 `passwordReset.json`、并尝试发送邮件（未配置 SMTP 时降级为控制台日志打印链接）。
- `POST /api/reset-password`（公开）：请求体 `{token, password}`，校验 token 未使用且未过期、新密码至少 6 位，使用 `crypto.scrypt` 重新生成 `passwordHash` 和 `salt`，标记同一用户全部未使用 token 为 `usedAt`。
- `POST /api/me/email`（登录态）：请求体 `{email}`，为当前登录用户设置或更新邮箱，做格式校验和跨账号唯一性校验。
- `POST /api/register` 改造：邮箱必填、格式正确、跨账号唯一；用户记录写入 `email` 字段。

### 13.4 前端交互
- 注册/登录表单顶部新增邮箱输入框，邮箱仅在注册时必填；登录时只用用户名/密码，邮箱框可用于触发"忘记密码"。
- 表单底部新增"忘记密码？"链接，点击时读取邮箱框的值发送 `POST /api/forgot-password`，并在状态栏显示统一文案。
- 页面初始化时如果 URL 含 `?reset=<token>` 查询参数，显示重置密码面板（新密码 + 确认密码两个输入框）；提交成功后清除 URL 参数并提示用户使用新密码登录。
- 登录后如果 `state.currentUser.email` 为空，显示一张"请补全邮箱"卡片，提供保存或暂不补全两个选项；该卡片只在当前会话内可暂不补全，下次登录仍会再次出现。
- 管理员图书管理区顶部新增"+ 新建图书"按钮，点击后在管理列表上方弹出新建图书表单（书名、作者、简介、封面预览、自动抓取按钮、取消按钮、保存按钮）；"自动抓取"按钮会调用 `POST /api/book-enrich`，将简介和封面预览填入表单；管理员可在抓取前后手动调整任意字段；保存调用 `POST /api/books`，成功后刷新图书列表和管理区。

### 13.5 SMTP / 环境变量
- `SMTP_HOST`、`SMTP_PORT`、`SMTP_USER`、`SMTP_PASS` 全部配置后启用真实邮件发送；可选 `SMTP_FROM`（默认与 `SMTP_USER` 相同）、`SMTP_SECURE`（`true` 或端口为 465 时启用 TLS）。
- 未配置时密码重置邮件不真发，服务器控制台会打印重置链接，方便本地调试；不暴露给客户端。
- `APP_BASE_URL` 用于生成邮件中的重置链接，默认是 `http://${HOST}:${PORT}`。
- SMTP 凭据通过环境变量传入，仓库内不保存任何真实密码；`.env` 已被 `.gitignore` 屏蔽。

### 13.6 安全约束
- 重置 token 在 `passwordReset.json` 中只存 sha256 哈希；原始 token 仅在邮件正文和服务器日志中出现一次。
- 重置 token 一次性使用，TTL 30 分钟；密码重置成功后同一用户所有未使用 token 一并失效。
- `POST /api/forgot-password` 不泄漏邮箱存在性，未注册邮箱也返回成功文案，仅在服务器日志中区分。
- 远程封面下载限制内容类型必须是 `image/*`，最大 5MB，30 秒超时，最多 4 次重定向；下载豆瓣图片时自动附加 `Referer: https://book.douban.com/`，下载失败降级为默认封面不阻塞创建。
- 邮箱唯一性校验：注册、`POST /api/me/email` 都校验跨账号唯一，避免找回密码歧义。
- 豆瓣搜索和详情页需要正常网络访问；抓取或解析失败时前端仍允许管理员手动填写简介和封面地址。
