概念 1/3 · 你现在的存储方式

你现在用的是「文件型数据库」

你的项目把数据存成了 4 个 JSON 文本文件，放在 data/ 文件夹里：

文件存什么结构
book.json5 本图书数组，每项有 id/title/author/description/cover
vote.json投票记录对象，按 bookId 存 count 和 records 列表
user.json注册用户数组，含密码哈希、salt、isAdmin
settings.json投票规则单个对象，voteMode 等

后端 server.js 每次都用 fs.readFile / fs.writeFile 读写这些文件。

⚠️ 痛点：两个人同时投票时，可能「同时读、同时写」，后写的覆盖前写的 → 丢票。文件还不好做查询、统计、备份。

概念 2/3 · Supabase 是什么

Supabase = 云上的 PostgreSQL + 一套工具

数据库（Database）：专门存数据的软件，支持多人同时安全读写、复杂查询、自动备份。

PostgreSQL：业界最流行的开源数据库之一（简称 Postgres）。

Supabase：把 Postgres 搬到云上，帮你托管，还附赠了网页管理后台、自动生成的 API、用户认证等。你不用自己装数据库、不用买服务器。

类比：JSON 文件像「记在记事本上的 Excel」；Supabase 像「公司共用的、有权限和锁的在线数据库系统」。

它有免费额度，做这个练习项目完全够用。

概念 3/3 · 为什么要迁移

迁移后能解决什么

现在 · JSON 文件

  • 并发投票可能丢数据
  • 统计/排序要自己写代码
  • 数据和代码混在一起，不好备份
  • 多台服务器无法共享数据

迁移后 · Supabase

  • 数据库帮你处理并发，不丢票
  • 一条 SQL 就能统计票数
  • 数据独立托管，自动备份
  • 网页后台可视化看数据、改数据

本教程的目标：数据搬进 Supabase，后端代码从「读写文件」改成「读写数据库」，前端完全不用动。

第 2 步 · 注册账号

注册 Supabase 账号

  1. 浏览器打开 https://supabase.com
  2. 点右上角 Start your project（开始你的项目）
  3. 用 GitHub 账号或邮箱登录（推荐 GitHub，最省事）
  4. 首次登录会让你创建一个 Organization（组织），名字随便填，比如 my-org，选 Free 免费套餐

✅ 你应该看到：登录后进入一个叫 Dashboard（仪表盘）的控制台页面，左边是菜单，中间提示你「New project」。

第 2 步 · 创建项目

新建一个项目（Project）

  1. 点 New project

  3. Database Password（数据库密码）：点 Generate 自动生成一个，务必复制保存到记事本，以后连数据库要用

第 2 步 · 看懂新版 Key

Key 换体系了：用哪一个？

旧教程里的 anon / service_role 已被新 Key 取代（旧的仍能用，但官方计划 2026 年底弃用）：

新 Key对应旧 Key用途本项目用？
Publishable key
sb_publishable_...		anon public前端 / 客户端❌
Secret key
sb_secret_...		service_role🔒 后端 / 服务端✅ 用这个

我们这个项目所有数据库操作都在后端，所以复制 Secret key（sb_secret_...），填到后面 .env 的 SUPABASE_SERVICE_KEY 里即可。

🔒 Secret key 权限极大：绝不能泄露、绝不能放进前端、绝不能提交到 Git。找不到新 Key 时，可在 API Keys 页点 Create new API keys 生成。

第 3 步 · 设计表

4 个 JSON 文件 → 4 张表

数据库里的「表（Table）」就像一张 Excel 表格：列（字段）是表头，行是一条条记录。

book.json→books 表 user.json→users 表
vote.json→votes 表（重点改造） settings.json→settings 表

最大变化在 votes：原来一本书的所有投票挤在一个对象里；数据库里我们让「一次投票 = 一行」，更清晰、不会丢、好统计。票数用 COUNT() 现算。

第 3 步 · 表结构

books 表

字段类型说明
idtext 主键沿用原来的 book-001
titletext书名
authortext作者
descriptiontext简介
covertext封面地址（可空）

主键（Primary Key）：每行的唯一身份证，不能重复。这里用书的 id。

第 3 步 · 表结构

users 表

字段类型说明
idtext 主键原来的 user-xxxx
usernametext 唯一用户名，不可重复
password_hashtext密码哈希（不存明文）
salttext加盐值
created_attimestamptz注册时间
is_adminboolean是否管理员，默认 false

💡 字段名用下划线小写（password_hash）是数据库习惯。代码里 JS 是 passwordHash，迁移时做个对应即可。

第 3 步 · 表结构（重点）

votes 表：一次投票一行

字段类型说明
idbigint 自增主键数据库自动编号
book_idtext 外键投给哪本书 → 关联 books.id
usernametext谁投的
voted_attimestamptz投票时间

外键（Foreign Key）：让 book_id 必须是 books 表里真实存在的书，防止脏数据。

票数不再单独存：SELECT count(*) FROM votes WHERE book_id='book-001' 随时现算，永远准确。

第 3 步 · 表结构

settings 表：只有一行

字段类型说明
idint 主键固定为 1（永远只有一行）
vote_modetextsingle_per_book 或 single_total
updated_attimestamptz最后修改时间
updated_bytext谁改的

这张表存全局配置，用一个固定 id=1 的行来保存当前投票规则。

第 3 步 · 建表 SQL

一次性建好 4 张表

左侧菜单点 SQL Editor → New query，粘贴下面全部代码，点 Run：

create table books (
  id text primary key,
  title text not null,
  author text not null,
  description text default '',
  cover text default ''
);

create table users (
  id text primary key,
  username text unique not null,
  password_hash text not null,
  salt text not null,
  created_at timestamptz default now(),
  is_admin boolean default false
);

create table votes (
  id bigint generated always as identity primary key,
  book_id text references books(id),  -- 外键
  username text not null,
  voted_at timestamptz default now()
);

create table settings (
  id int primary key default 1,
  vote_mode text not null default 'single_per_book',
  updated_at timestamptz,
  updated_by text
);

✅ 你应该看到：右下角提示 Success. No rows returned。在左侧 Table Editor 里能看到 4 张空表。

第 4 步 · 导入数据

把现有 JSON 数据搬进去

三类数据，三种做法，都不难：

  • books / users：结构和表几乎一样，直接写 insert SQL 插入。
  • votes：要把 vote.json 里每条 record 拆成单独一行。
  • settings：插入固定的一行 id=1。

数据量很小（5 本书、2 个用户），手写 SQL 最直观。下一页给你现成模板。

第 4 步 · 导入数据

插入数据的 SQL（在 SQL Editor 运行）

-- 图书（按你的 book.json 逐条填）
insert into books (id, title, author, description) values
('book-001', '深入浅出 Node.js', '朴灵', '介绍 Node.js 原理...'),
('book-002', 'JavaScript 高级程序设计', 'Nicholas C. Zakas', '...');
-- 其余 book-003~005 照此补全

-- 用户（password_hash / salt 从 user.json 原样复制）

('user-xxxx', 'wenfeng.zhu@gmail.com', '原哈希', '原salt', true);

-- 投票：vote.json 里每条 record 写一行
insert into votes (book_id, username, voted_at) values
('book-001', 'wenfeng.zhu@gmail.com', '2026-05-13T01:58:39.544Z');

-- 投票规则

⚠️ 旧 vote.json 里有些书 count 大于实际 records（如 book-002 count=3 但 records 为空）。迁移时以真实 records 为准，那些「虚高」的票就自然不计了。

第 5 步 · 改造代码

② 新建数据库连接 db.js

新建 db.js，统一创建一个 Supabase 客户端供全项目使用：

require("dotenv").config();
const { createClient } = require("@supabase/supabase-js");

// Node < 22 没有内置 WebSocket，用 ws 补上
if (!globalThis.WebSocket) {
  globalThis.WebSocket = require("ws");
}

const supabase = createClient(
  process.env.SUPABASE_URL,
  process.env.SUPABASE_SERVICE_KEY,  // 后端用 Secret key
  { auth: { persistSession: false } }
);

module.exports = { supabase };

之后在 server.js 顶部引入它，就能用 supabase 读写数据库了。

const { supabase } = require("./db");
第 5 步 · 改造代码（对比）

③ 查询图书 + 票数

改造前 · 读文件

const books =
  await readBooks();
const votes =
  await readVotes();
// 自己遍历算票数

改造后 · 查数据库

// 图书：字段名和原来一样
const { data: books } =
  await supabase
    .from("books")
    .select("*");

// 投票：取所有行，再按
// book_id 聚合成票数+记录
const { data: rows } =
  await supabase
    .from("votes")
    .select("book_id, username, voted_at");

.from("表名").select("*") = 「从这张表查所有列」。投票行用 JS 按 book_id 累加票数、收集记录，拼成和旧代码一样的结构，前端拿到的格式完全不变。

第 5 步 · 改造代码（对比）

④ 投票接口 POST /api/votes

改造前

const votes =
  await readVotes();
votes[bookId] = nextEntry;

  votesFile, votes);
// 读改写，可能并发丢票

改造后

// 1. 按投票规则检查
//    （是否已投过该书 / 任意书）
assertCanVote(
  settings, votes,
  bookId, username);

// 2. 插入一行新投票

第 6 步 · 安全

密钥安全三条铁律

  1. 密钥放 .env，代码里用 process.env.XXX 读取，绝不硬编码。
  2. .env 写进 .gitignore，永远不提交到 Git / GitHub。
  3. Secret key（sb_secret_...）只在后端用，绝不出现在 public/ 前端代码里。前端要用就用 Publishable key。

关于 RLS（行级安全）

RLS（Row Level Security）：Postgres 的一道门禁，控制「谁能读/写哪些行」。

本项目所有数据库操作都走后端（用 Secret key，会绕过 RLS），所以你这一版保持默认即可。等以后让前端直连数据库时，再回来学 RLS。

第 7 & 8 步 · 验证与排坑

验证 + 常见报错

✅ 怎么确认成功

  • npm start 启动，打开网站投一票 → 回 Supabase Table Editor 刷新 votes 表，应能看到新增的一行。
  • 退出登录、用别人账号投，票数实时变化，规则（不能重复投）仍然生效。

🐞 最容易踩的坑

报错 / 现象原因 & 解决
supabaseUrl is required.env 没加载或拼错 → 确认 require("dotenv").config() 在最上面
insert 报 foreign key 违反votes 的 book_id 在 books 表里不存在 → 先导 books 再导 votes
duplicate key / uniqueusername 或 id 重复插入 → 检查是否重复运行了导入 SQL
data 为 null / 空多半是 RLS 拦了 → 确认后端用的是 Secret key（sb_secret_...）
列名报错JS 的 passwordHash ↔ 表的 password_hash 没对应上
