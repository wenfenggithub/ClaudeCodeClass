# 图书投票应用 MVP 规格说明

## 1. 项目目标

开发一个网页版本的图书投票应用，用于展示约 5 本候选图书，并允许已登录用户为喜欢的图书投票。

当前阶段目标是完成 MVP 版本，重点验证核心投票流程是否可用：

- 用户可以看到图书列表。
- 用户可以看到每本书的当前票数。
- 用户可以注册账号并登录。
- 用户登录后可以点击投票按钮。
- 未登录用户不能投票。
- 投票成功后，页面票数立即更新。
- 后端使用 Node.js 提供接口。
- 投票数据暂时使用本地 `book.json` 文件存储。
- 用户数据暂时使用本地 `users.json` 文件存储。

当前版本只包含基础注册和登录能力，不包含复杂权限系统、找回密码、验证码、重复投票限制等复杂功能。

## 2. MVP 功能范围

### 必须包含

- 图书列表展示
- 每本书展示基础信息
- 每本书展示当前票数
- 每本书提供投票按钮
- 用户注册
- 用户登录
- 用户登录状态识别
- 页面显示当前登录用户
- 用户退出登录
- 未登录用户不能投票
- 点击投票后更新对应图书票数
- 前端从后端获取图书和票数数据
- 后端读取 `book.json` 中的数据
- 后端将投票后的最新票数写回 `book.json`
- 后端读取和写入 `users.json` 中的用户数据
- 投票失败时给出基础错误提示
- 注册或登录失败时给出基础错误提示

### 暂不包含

- 复杂用户权限管理
- 管理员角色
- 找回密码
- 邮箱或短信验证
- 验证码
- 第三方登录
- 每个用户只能投一次的限制
- 后台管理页面
- 数据库
- 图书搜索
- 图书分类
- 排行榜复杂筛选

## 3. 页面功能说明

页面主要由用户登录状态区域和图书投票列表组成。

用户登录状态区域建议包含：

- 未登录时展示登录入口和注册入口
- 已登录时展示当前用户名
- 已登录时提供退出登录按钮
- 未登录投票时提示用户先登录

每本书建议展示以下信息：

- 图书名称
- 作者
- 简短描述
- 当前票数
- 投票按钮

页面加载时，前端请求后端接口获取图书列表和票数数据。获取成功后，将图书数据渲染到页面中。

页面加载时，前端还需要检查当前用户登录状态。若用户未登录，投票按钮可以保持可见，但点击时应提示先登录；也可以直接置为禁用状态并展示登录提示。

用户登录后点击某本书的投票按钮，前端向后端发送投票请求。后端验证登录状态后处理投票，成功后返回最新数据，前端更新页面中的票数。

## 4. 投票交互流程

1. 用户打开网页。
2. 前端请求后端图书列表接口。
3. 后端读取 `book.json` 文件。
4. 后端返回图书列表和当前票数。
5. 前端检查当前用户登录状态。
6. 前端渲染图书列表和登录状态。
7. 未登录用户点击投票按钮时，前端提示先登录，不发送投票请求或发送后由后端返回 401。
8. 已登录用户点击某本书的投票按钮。
9. 前端向后端发送投票请求，携带被投票图书的 ID 和登录凭证。
10. 后端验证登录状态。
11. 后端根据图书 ID 找到对应图书。
12. 后端将该图书票数加 1。
13. 后端把更新后的数据写回 `book.json`。
14. 后端返回投票成功结果和最新票数。
15. 前端更新页面展示。

## 5. 注册和登录交互流程

### 注册流程

1. 用户打开注册表单。
2. 用户输入用户名和密码。
3. 前端向后端发送注册请求。
4. 后端读取 `users.json`。
5. 后端检查用户名是否已存在。
6. 若用户名不存在，后端创建新用户并写回 `users.json`。
7. 后端返回注册成功结果。
8. 前端提示注册成功，并引导用户登录或直接进入登录状态。

### 登录流程

1. 用户打开登录表单。
2. 用户输入用户名和密码。
3. 前端向后端发送登录请求。
4. 后端读取 `users.json`。
5. 后端检查用户名和密码是否匹配。
6. 登录成功后，后端返回当前用户信息和登录凭证。
7. 前端保存登录凭证，并展示当前用户名。
8. 用户点击退出登录时，前端清除登录凭证并恢复未登录状态。

## 6. Node.js 后端设计思路

后端使用 Node.js 提供 HTTP API，负责读取图书数据、处理投票请求、写入最新票数，并处理基础用户注册和登录。

建议后端职责划分如下：

- 静态资源服务：提供前端页面、CSS、JavaScript 等文件。
- 数据读取：从 `book.json` 读取当前图书和票数信息。
- 数据写入：投票成功后，将最新数据写回 `book.json`。
- 用户数据读取：从 `users.json` 读取用户信息。
- 用户数据写入：注册成功后，将新用户写回 `users.json`。
- API 接口：提供获取图书列表、提交投票、注册、登录、获取当前用户和退出登录的接口。
- 登录验证：投票接口需要验证用户是否已登录。
- 错误处理：处理文件读取失败、图书不存在、请求参数错误、用户名已存在、密码错误、未登录投票等情况。

MVP 阶段可以使用 Node.js 原生能力或 Express 框架。若项目希望更容易扩展，推荐使用 Express。

基础登录凭证可以使用简单 token 方案：登录成功后后端返回 token，前端保存在 `localStorage` 或 `sessionStorage`，后续请求在 `Authorization` 请求头中携带。MVP 阶段可以先满足登录状态识别；后续再替换为更完整的 Session、JWT 或数据库方案。

密码不应明文保存。即使是 MVP，也建议保存密码哈希，例如使用 `passwordHash` 字段。若只是本地学习演示，也需要在文档和代码中明确这不是生产级认证方案。

## 7. book.json 数据结构设计

`book.json` 用于保存图书信息和票数。

建议数据结构如下：

```json
[
  {
    "id": "book-1",
    "title": "示例图书 1",
    "author": "作者 1",
    "description": "这是第一本候选图书的简介。",
    "votes": 0
  },
  {
    "id": "book-2",
    "title": "示例图书 2",
    "author": "作者 2",
    "description": "这是第二本候选图书的简介。",
    "votes": 0
  }
]
```

字段说明：

- `id`：图书唯一标识，用于投票时定位具体图书。
- `title`：图书名称。
- `author`：作者。
- `description`：图书简介。
- `votes`：当前票数。

## 8. users.json 数据结构设计

`users.json` 用于保存基础用户信息。

建议数据结构如下：

```json
[
  {
    "id": "user-1",
    "username": "demo",
    "passwordHash": "hashed-password",
    "createdAt": "2026-05-12T00:00:00.000Z"
  }
]
```

字段说明：

- `id`：用户唯一标识。
- `username`：用户名，注册时需要保证唯一。
- `passwordHash`：密码哈希，不建议保存明文密码。
- `createdAt`：用户创建时间。

## 9. API 接口设计

### 获取图书列表

请求：

```text
GET /api/books
```

用途：

获取所有图书信息和当前票数。

成功响应思路：

```json
{
  "success": true,
  "data": [
    {
      "id": "book-1",
      "title": "示例图书 1",
      "author": "作者 1",
      "description": "这是第一本候选图书的简介。",
      "votes": 0
    }
  ]
}
```

### 提交投票

请求：

```text
POST /api/books/:id/vote
```

用途：

为指定 ID 的图书增加 1 票。该接口需要登录后才能调用。

请求头思路：

```text
Authorization: Bearer <token>
```

成功响应思路：

```json
{
  "success": true,
  "data": {
    "id": "book-1",
    "votes": 1
  }
}
```

失败响应思路：

```json
{
  "success": false,
  "message": "Book not found"
}
```

未登录失败响应思路：

```json
{
  "success": false,
  "message": "请先登录后再投票"
}
```

### 用户注册

请求：

```text
POST /api/auth/register
```

请求体思路：

```json
{
  "username": "demo",
  "password": "123456"
}
```

用途：

创建一个新用户。

成功响应思路：

```json
{
  "success": true,
  "data": {
    "id": "user-1",
    "username": "demo"
  }
}
```

失败响应思路：

```json
{
  "success": false,
  "message": "用户名已存在"
}
```

### 用户登录

请求：

```text
POST /api/auth/login
```

请求体思路：

```json
{
  "username": "demo",
  "password": "123456"
}
```

用途：

验证用户身份，并返回登录凭证。

成功响应思路：

```json
{
  "success": true,
  "data": {
    "token": "login-token",
    "user": {
      "id": "user-1",
      "username": "demo"
    }
  }
}
```

失败响应思路：

```json
{
  "success": false,
  "message": "用户名或密码错误"
}
```

### 获取当前用户

请求：

```text
GET /api/auth/me
```

用途：

根据登录凭证获取当前登录用户信息。

成功响应思路：

```json
{
  "success": true,
  "data": {
    "id": "user-1",
    "username": "demo"
  }
}
```

未登录响应思路：

```json
{
  "success": false,
  "message": "未登录"
}
```

### 退出登录

请求：

```text
POST /api/auth/logout
```

用途：

退出当前登录状态。若使用前端本地 token，退出登录可以由前端清除 token 完成；后端接口可作为统一交互入口，返回成功即可。

成功响应思路：

```json
{
  "success": true
}
```

## 10. 错误处理思路

MVP 阶段需要处理以下基础错误：

- `book.json` 文件不存在：后端返回服务器错误，并提示数据文件缺失。
- `book.json` 内容不是合法 JSON：后端返回服务器错误，并提示数据格式异常。
- 投票时图书 ID 不存在：后端返回 404。
- 未登录用户投票：后端返回 401，并提示先登录。
- 文件写入失败：后端返回服务器错误，并提示投票保存失败。
- `users.json` 文件不存在：后端可以初始化为空数组，或返回服务器错误。
- `users.json` 内容不是合法 JSON：后端返回服务器错误，并提示用户数据格式异常。
- 注册时用户名为空：后端返回 400。
- 注册时密码为空或过短：后端返回 400。
- 注册时用户名已存在：后端返回 409。
- 登录时用户名不存在：后端返回 401。
- 登录时密码错误：后端返回 401。
- 登录凭证无效或过期：后端返回 401。
- 前端请求失败：页面展示简单错误提示，例如“操作失败，请稍后重试”。

后续如果加入高并发或多用户场景，需要考虑文件写入冲突和重复投票控制。

## 11. 后续可扩展功能

后续版本可以考虑增加：

- 每个用户只能为每本书投一次
- 每个用户只能选择一本书投票
- 用户资料编辑
- 密码修改和找回密码
- 更安全的 Session 或 JWT 方案
- 密码强度规则和验证码
- 投票排行榜
- 图书新增、编辑、删除功能
- 后台管理页面
- 数据库存储，例如 SQLite、PostgreSQL 或 MongoDB
- 投票截止时间
- 投票结果图表
- 防刷票机制
- 更完整的错误日志
- 单元测试和接口测试

## 12. MVP 验收标准

MVP 完成后，应满足以下条件：

- 打开网页可以看到约 5 本图书。
- 每本图书都有清晰的投票按钮。
- 每本图书都能显示当前票数。
- 用户可以注册账号。
- 用户可以登录账号。
- 登录后页面显示当前用户名。
- 用户可以退出登录。
- 未登录用户不能投票。
- 已登录用户点击投票后，对应图书票数加 1。
- 刷新页面后，票数仍然保持更新后的结果。
- 后端数据来自 `book.json`。
- 用户数据来自 `users.json`。
- 投票后的数据会写回 `book.json`。
